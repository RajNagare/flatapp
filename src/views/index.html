Gus is a {{ gusDescription }}. Built on a simple route > controller > view architecture, applications built with Gus can be scaled without being re-written.

Gus is a simple, vanilla PHP stack that requires no database and has all the tools you need to get your application started. The goal was to have a platform that made it easy to crank out web apps.

Install That Puppy

Gus gets you going fast by making setup a breeze and having common dependecies already included:

  • Templating by Twig
  • UI Development by jQuery and JiveJS
  • Responsive UI by CardinalCSS
  • HTML 5 Feature Detection by Modernizr

Users visit web apps by a URL or a route. Each route has route-specific logic which is handled by a controller. The controller display data on template or a view.

How It Works

Every request to Gus goes to the Routing Controller. Here, the application searches the controller/ directory, trying to match a PHP file by filename with the supplied route.

Gus askes: Does controller/{route}.php exist? If so, that is our route. If not, display a 404

If the controller exists, Gus looks in the views/ directory for a similarly named HTML file like so: views/{route}.html

jQuery, JiveJS, Cardinal CSS, Modernizr, and Twig included in every application. Easily keep them up to date or add new ones through git submodules.

Libraries

The /lib directory is where all 3rd party libraries go. If you want to integrate things like database object interaction, mailer wrappers, or other projects, this is the place to do it.

Git Submodules

Instead of downloading tarballs, you can use git submodules to add a package managment-like experience for maintaining your libaries.

Keep only what you want

Prefer vinylbowl.js over ironicglasses.js? You're not locked into any library, so customize at your will.

Installation should be a breeze

Update The Config

Update config.ini in the applications root directory to match your server's absolute path for the application. Feel free to use this config file for other things too. We use INI files for configs, so learn more about it @phpnet's ini parsing page.

Permissions for View Cache

Make sure that views/cache/ is writeable by apache (or the like).

VHOST Block / HTAccess Setup

Make sure .htaccess is in the application root directory. Apache vhost blocks also need to be correctly configured to use mod_rewrites on a vhost block level.
